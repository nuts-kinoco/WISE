# Jules QA Sprint 30 — テスト依頼プロンプト集

Deep Review Sprint 30（P1: WiseDbContext直接注入の是正 Phase1-4／P2: メタデータ収集の終了条件再設計）の
リファクタリング検証スプリント。**今回の変更は「レスポンス形状を変えずに内部実装だけ差し替えた」もの
がほとんどのため、目視での回帰確認が最重要**。ロジックを書き換えたコントローラー・UseCaseすべてで、
リファクタ前と同じ挙動になっているかを重点的に見てほしい。

Julesは並行実行可能なため、**Prompt A〜Eを個別セッションとして同時に投入すること**。
書き込みを伴うテスト（Duplicates解消・トリアージ保存・Collection操作・Job操作）は、他プロンプトと
同時に同一作品/同一ジョブを触らないよう注意書きを添えてある。

前提として、まず `Document/Review_P1_DbContextRefactoring.md`, `Document/Review_P2_PipelineExitPolicy.md`,
`Document/Review_P3_EfCoreAudit.md` を読み、何がどう変わったかを把握してから着手すること。
起動: API = `dotnet run --project src/WISE.Api`（port 5162）、Web = `npm run dev`（`src/wise-web/`, port 3000）。

対象コミット範囲: `2b9a877`〜`258bb69`（P2実装 〜 P1 Phase4実装）

   引き続き正しく動作するか確認する。

## Prompt B — コレクション・リーダー・アセット配信の回帰テスト（P1 Phase3）

対象: `CollectionsController.cs`, `ReaderController.cs`, `AssetsController.cs`

### コレクション
1. 新しいコレクションを作成し、任意の作品を1件以上追加できるか確認する。
2. コレクション一覧（`/collections`）でカバー画像・作品数が正しく表示されるか確認する
   （`ICollectionsQueryService`のカバー画像解決ロジックが変わっているため重点確認）。
3. コレクション詳細ページで、追加した作品のタイトル・出演者等のメタデータが正しく表示されるか確認する
   （`WorkItemMapper.Map`の呼び出し位置がQueryServiceからコントローラーに変わっているため、
   表示内容が変わっていないか重点確認）。
4. 同じ作品を重複追加しようとして拒否される（409）ことを確認する。
5. コレクションから作品を削除し、コレクション自体も削除できるか確認する。

### コミックリーダー
6. コミック/画像集作品の詳細ページから「読む」を開き、ページ一覧取得・ページ画像の表示が
   正常に行われるか確認する（`IReaderQueryService`経由になった）。
7. ページ送り・戻りが正常に動作し、キャッシュされたページの再表示が速いか確認する。

### アセット配信（動画キャッシュ）
8. 動画作品を再生し、通常再生・シークともに問題なく動作するか確認する
   （`AssetsController`の`GetContent`が`IAssetsQueryService`経由になったが、
   `VideoStreamCache`のRange処理ロジック自体は変更していない）。

※他プロンプトと同時実行の場合、このプロンプト専用に新規作成したコレクションのみ削除すること。

## Prompt C — ジョブキュー・重複解消の回帰テスト（P1 Phase4・重要）

対象: `JobsController.cs`→`IJobsQueryService`/`JobUseCase`, `DuplicatesController.cs`→`IDuplicatesQueryService`/`DuplicateResolveUseCase`

**このプロンプトは今回のリファクタで最もロジックが変わった箇所（`DuplicateResolveUseCase`の
トランザクション化）を含むため、特に重点的に検証してほしい。**

### ジョブキュー
1. 詳細ページの「再スキャン」ボタンからメタデータ再取得ジョブをキューに入れ、ジョブキューパネルに
   反映され、進捗が表示され、完了後にパネルから消えるか確認する。
2. トリアージページ等から複数作品を選択して一括再スキャン（`fetchmetadata/batch`）を実行し、
   正しい件数がキューに入るか確認する。
3. 実行中のジョブをキャンセルできるか、失敗したジョブをリトライできるか確認する
   （`/jobs`ページの該当ボタン）。
4. 完了・失敗・キャンセル済みジョブの一括削除（「クリア」ボタン等）が正しく動作するか確認する。

### 重複解消（★最重要）
5. 重複グループ（同一識別子で複数登録された作品）を用意し、一覧に正しく表示されるか確認する。
6. 重複のうち1つに評価・お気に入り・メモ・ユーザータグを設定した状態で重複解消を実行し、
   **「保持する」側にそれらのユーザーデータが正しく引き継がれる**か確認する
   （マージロジック自体は変更していないが、実行順序が「全部の対象を一括処理」に変わっている）。
7. **3件以上の重複グループ**で重複解消を実行し、全ての削除対象が正しく削除され、保持した作品には
   正しくデータがマージされているか確認する（旧実装はループ内で1件ずつDB確定していたが、今回
   全体を1トランザクションにまとめたため、3件以上での挙動に差が出ていないか重点確認）。
8. 「ファイルも削除」オプションを付けて重複解消を実行し、物理ファイルが削除される、あるいは
   ファイルがロック中で削除できない場合の挙動（DBの重複解消自体は成功しファイル削除失敗のみ
   報告される想定）を確認する。
9. 存在しない作品IDを指定した場合や、`deleteWorkIds`が空の場合にエラーが返るか確認する。

※他プロンプトと同時実行の場合、ジョブ操作・重複解消は他プロンプトが操作中の作品/ジョブに
触れないよう、テスト専用に用意した作品で実施すること。

## Prompt D — メタデータ収集パイプラインの回帰テスト（P2）

対象: `MetadataService.cs`（終了条件のProvider特性ベース化）, `Fc2MetadataProvider.cs`, `Fc2AltMetadataProvider.cs`

**変更点**: 早期終了の判定が「識別子がFC2で始まるか」から「実際に成功したProviderが供給可能と
宣言しているフィールド」に変わった。挙動が同じであることの確認が目的。

1. FC2品番（`FC2-PPV-XXXXXXX`）でメタデータ再取得を実行し、ログ（`[WISE.Metadata]`プレフィックス）で
   `Text fields satisfied at Priority=...`が**Fc2/Fc2Altの実行直後**に出ていることを確認する
   （Actress/Makerを待たずに早期終了しているか）。
2. 通常のAV品番（FC2以外）でメタデータ再取得を実行し、Title/Actress/Makerが揃うまで複数Tierが
   実行されることをログで確認する（早期終了しすぎていないか）。
3. コミック作品でメタデータ再取得を実行し、Title/Author/Circleが揃った時点で早期終了することを
   ログで確認する。
4. 上記いずれのケースでも、カバー画像取得のフォールバック（Phase2、`[CoverFallback]`ログ）が
   従来通り動作し、カバー画像が正しく取得されているか確認する。
5. 全Providerが失敗するような不正な品番（存在しない識別子等）でメタデータ取得を実行し、
   エラーにならず、あるいは適切に失敗として処理される（無限ループ等が発生しない）ことを確認する
   （早期終了判定に「成功Providerが0件なら早期終了しない」ガードを追加したため、この境界を確認）。

## Prompt E — 履歴・監視フォルダの回帰テスト（P1 Phase1）＋ 横断確認

対象: `HistoryController.cs`→`IHistoryQueryService`, `WatchFoldersController.cs`→`WatchFolderUseCase`

### 履歴
1. 履歴ページ（`/history`）で、直近の操作履歴（インポート完了・メタデータ取得等）が正しい
   日本語ラベル・アイコンで表示されるか確認する。
2. 履歴の各項目から対象作品への遷移（`targetIdentifier`/`targetId`があるもの）が正しく機能するか確認する。

### 監視フォルダ
3. インポートページ（`/import`）で監視フォルダを追加できるか確認する。
4. 追加した監視フォルダの有効/無効切り替え（トグル）が正しく機能するか確認する。
5. 同じパスを重複登録しようとして拒否される（409）ことを確認する。
6. 監視フォルダを削除できるか確認する。

### 横断確認（アプリ全体）
7. サーバーを起動した状態でブラウザのDevTools Consoleを開き、上記A〜E全体を通して操作している間、
   500エラー・未処理例外・CORSエラー等が一切出ていないか確認する。
8. `dotnet run --project src/WISE.Api`の標準出力に、今回のリファクタに起因するとみられる
   警告・例外スタックトレースが出ていないか確認する（特に`InvalidOperationException`が出ていないか。
   これは今回P3監査で修正した「DbContextへの並行クエリ」系のバグの典型的な症状のため）。
